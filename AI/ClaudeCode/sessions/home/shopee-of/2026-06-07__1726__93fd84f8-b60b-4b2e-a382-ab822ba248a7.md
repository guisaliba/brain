Good — lots of important nuance here. Let me address each point:

---

**1. apply-coupon + ships_free logic**

You're right — the plan oversimplifies. The step's error classification depends on whether a coupon was *needed*. The correct logic:

1. Inspect shipping cost area first (before touching coupon). If shipping is already R$0,00/free → product ships free natively. Set `products.shipsFree = true`. **No coupon needed.** Step succeeds regardless of coupon availability.
2. If shipping is NOT free → look for "Frete Gratis" coupon. If found → apply it, set `couponDiscount`. Step succeeds.
3. If shipping is NOT free AND no coupon available → *this* is the `coupon_unavailable` recoverable error. The product requires shipping cost and we can't offset it.

So `coupon_unavailable` is only an error when the product doesn't ship free natively. When it does, the step logs "no coupon applied — product ships free" and continues normally.

---

**2. submit-email step**

Looking at the current code (lines 268-273 of `run-checkout.ts`): after clicking "Fazer pedido", Shopee sometimes presents an email confirmation screen — it asks for the email associated with the Shopee account placing the order (`ctx.account.email`). It's a Shopee-side verification step that appears intermittently. If the account has an email on file, the code types it and clicks "Confirmar". If Shopee doesn't ask for it (no email input visible), the step is a no-op. It's the Shopee account's email, not the customer's.

---

**3. CAPTCHA detection scope**

You're absolutely right — limiting detection to two points creates blind spots. Shopee can spawn CAPTCHAs at any interaction. The safer design: a `withCaptchaGuard(page, ctx, action)` wrapper that calls `detectCaptcha()` after every page navigation and every significant interaction. Every step that triggers navigation or a page-level action goes through this wrapper. This catches CAPTCHAs at:
- Every `page.goto()`
- After "Comprar agora" / "Adicionar ao carrinho" clicks
- After "Finalizar compra"
- After address submission
- After "Fazer pedido"
- After email confirmation

Also: you building the 2Captcha resolver is noted. The plan should account for the `CaptchaResolver` interface being satisfied by a real resolver, not just the stub.

**4. ships_free scraping detail**

Here's the detailed flow for `apply-coupon.ts`:

1. **Inspect shipping cost display** — on the Shopee checkout page, locate the shipping cost section. Look for elements showing the shipping price (e.g. text matching `R$` near "Frete"/"Envio"/"Entrega"). Extract the numeric value.
2. **Determine if product ships free natively**:
   - If shipping cost shows R$0,00 or contains "Gratis"/"Grátis" WITHOUT any coupon applied → `shipsFree = true`. Update `products.shipsFree = true` in DB.
   - If shipping cost is > R$0,00 → product does NOT ship free natively.
3. **Coupon logic** (only if shipping is NOT free):
   - Look for "Frete Grátis" coupon button/badge in the checkout coupon area.
   - If visible → click to apply. Wait for shipping cost to update to R$0,00. Verify it changed. Record `couponDiscount`.
   - If not visible → `coupon_unavailable` error (recoverable).
4. **Success conditions**: shipping cost is zero (either natively or via coupon). The step records *how* free shipping was achieved (native vs coupon) in evidence.
5. **Error condition**: shipping is NOT free AND no coupon available.
6. **Fallback**: if scraping the shipping cost fails entirely (elements not found), fall back to the existing `products.shipsFree` value set manually. If that's also null, treat as "unknown" and proceed without coupon — let the human verify the final price at payment time.
